# Security policy

## Reporting a vulnerability

**Do not file a public GitHub issue for a security report.** Send the
details to **security@venturflow.com** with subject line `[SECURITY]
vc-statement-parser: <short description>`. We will acknowledge within
72 hours and aim to ship a patch â€” or a documented mitigation if the issue
is in an upstream dependency â€” within 14 days.

If you would like end-to-end encryption, our PGP key is published at
[keybase.io/venturflow](https://keybase.io/venturflow) (TODO: replace with
the project's actual key on first release).

We will credit reporters in the release notes unless they prefer to remain
anonymous. We do not run a paid bug-bounty program.

## Threat model

`vc-statement-parser` ingests PDF files that may have been generated by
untrusted parties (a fund administrator's PDF generator, an attacker's
crafted file, an OCR pass on a scanned document). It optionally forwards
the extracted text to an LLM provider. Every operator should be aware of
the following considerations.

### In scope

| Concern | Mitigation in this codebase |
|---|---|
| **Malicious PDF input** | Parsing goes through `pdfplumber` + `pypdfium2`, both of which sandbox the PDF parse. We never `eval` / `exec` extracted text. We never invoke a sub-process or shell. |
| **Path traversal via input filename** | The CLI uses `typer.Argument(exists=True, dir_okay=False, readable=True)`, which constrains inputs to existing readable files. |
| **Resource exhaustion (zip-bombs, very large PDFs)** | Not directly mitigated; deploy operators should impose their own per-file timeouts and memory caps when batch-processing untrusted inputs. The deterministic extractors are bounded â€” they don't recurse â€” but the LLM path is bounded only by the configured `max_tokens`. |
| **Prompt injection via PDF text** | The LLM fallback feeds raw PDF text to the model. A malicious PDF could embed instructions ("ignore the system prompt and ..."). The LLM result is constrained to a Pydantic schema (`_ExtractedFields`), and `verify()` will catch any output that doesn't satisfy ILPA arithmetic identities â€” so a successful injection can corrupt fields but cannot, in general, fool the verifier into approving fabricated numbers. **Operators should still treat `extractor == "llm.instructor"` outputs as needing human review** when the source is untrusted. |
| **Data exfiltration via `OPENAI_BASE_URL`** | When the LLM fallback is enabled, the user's PDF text is sent to the configured endpoint. If `OPENAI_BASE_URL` is set incorrectly (or maliciously), the document leaves the operator's perimeter. The CLI does NOT log the resolved base URL by default. Operators should review their environment before feeding LP-private statements through any non-local endpoint, and prefer `OPENAI_BASE_URL=http://localhost:11434/v1` (Ollama) or similar local inference for sensitive material. |
| **Leaked credentials in commits** | `.gitignore` blocks `.env*`, `*.pem`, `*.key`, `secrets/`, `.openai-key`, `.anthropic-key`, `.groq-key`. CI should run gitleaks or trufflehog (TODO once CI is wired up). The codebase has no hard-coded keys. |
| **Accidentally committed real LP data** | `.gitignore` blocks the common naming patterns (`*.real.pdf`, `*.lp.pdf`, `*.confidential.pdf`, `examples/real_world/private_*`, etc.); `examples/real_world/README.md` documents the provenance rule. |

### Explicitly out of scope

- **Side-channel attacks against the LLM provider.** If you trust your
  cloud provider's response, we trust their response. We do not validate
  TLS certificates beyond what `openai-python` already does.
- **Securing the operator's local file system.** If an attacker can write
  to your filesystem they can replace any input file we read.
- **Multi-tenant isolation.** This library is a single-process Python
  parser; there is no auth surface to harden.

## What this project deliberately does NOT do

- **No telemetry.** `vc-statement-parser` does not phone home, does not
  log usage to any third party, does not include analytics.
- **No automatic updates.** We will not silently change behavior of an
  installed version.
- **No outbound network calls except the explicitly-configured LLM endpoint.**
  Deterministic extractors are 100% offline.

## Hardening checklist for operators

If you are running this against real fund-administrator PDFs in production:

1. Run with `--no-llm` whenever the document type is recognised by a
   deterministic extractor â€” this guarantees no data leaves the host.
2. If the LLM fallback is required, prefer a local model:
   `OPENAI_BASE_URL=http://localhost:11434/v1` (Ollama) or LM Studio.
3. Pin dependencies in your downstream project; do NOT `pip install
   vc-statement-parser` without a version constraint. We follow semver.
4. Review the `verify()` report before treating any parse as trustworthy â€”
   a passed parse with `nav_roll_forward FAIL` is a bug, not data.
5. When ingesting attacker-controlled PDFs (e.g. an upload form), enforce
   timeouts and memory caps at the OS / container level.
